# 当前状态 PRD

更新时间：2026-04-16  
文档类型：现状 PRD / 基线说明书  
适用范围：当前仓库实现、`data/` 快照、已完成的真实部署与验收结果

## 1. 文档目的

这份 PRD 不描述理想终局，而是回答 3 个问题：

1. 这个项目现在到底是什么产品
2. 它已经真实具备了哪些能力
3. 下一阶段应该沿着什么方向继续做，而不是重做

## 2. 当前产品定位

当前项目的准确定位是：

- 一个面向低内存 VPS / LXC 节点的轻量代理节点控制面
- 更接近“轻量机场控制面 / 代理节点控制面”
- 已经明显超出“纯节点纳管平台”的范围

原因：

- 产品已经包含 `Node / AccessUser / ProxyProfile / NodeGroup / ConfigRelease / Share / Subscription / QR` 等业务对象
- 已经具备节点纳管、配置发布、订阅输出、系统账号/模板下发、业务链路建模等完整产品雏形
- 当前主要价值不再只是“看机器活没活着”，而是“让一批代理节点可被纳管、发布、验证、分享和运维”

结论：

- 对外描述不应继续以“通用多云控制平台”为主
- 当前最准确的说法应是“单机形态的轻量代理节点控制面”

## 3. 目标用户

当前阶段的核心用户不是终端订阅用户，而是运维侧操作者：

- 个人站长 / 小团队运维：统一纳管多台低配 VPS、LXC、落地机和入口机
- 代理节点运营者：批量发布用户、模板、节点线路与订阅
- 基础设施维护者：通过 SSH、模板、批量操作、探测结果处理节点异常

当前不是面向终端用户的自助后台，也不是多租户 SaaS 面板。

## 4. 当前要解决的问题

项目目前聚焦解决以下实际问题：

- 节点分散在多个厂商和多个地域，台账与状态分散
- 低配节点不适合安装重型 Agent，希望以 SSH / 一次性 bootstrap 为主
- 节点需要统一初始化、统一配置、统一发布接入用户
- 业务流量链路与管理链路需要分开建模，否则运维和分享都会混乱
- 需要从控制面持续观察节点健康，而不只是偶尔登录机器查看
- 需要把“节点 -> 发布 -> 分享链接 -> 订阅内容”串成闭环

## 7. 当前主链路

### 7.1 纳管链路

1. 操作者在节点上执行 bootstrap 命令
2. 节点上报基础事实与网络入口信息
3. 控制面创建或更新节点
4. 平台回传 SSH 公钥与初始化动作
5. 节点进入可继续接管与探测的状态

### 7.2 管理运维链路

1. 控制面根据 `management.*` 或 SSH 规则解析接管路径
2. 通过直连或 SSH 跳板进入目标节点
3. 执行初始化、系统用户、系统模板或批量操作
4. 将任务结果、输出摘要、节点状态写回控制面

### 7.3 业务发布链路

1. 创建或选择 `AccessUser`
2. 选择 `ProxyProfile`
3. 选择 `NodeGroup` 或节点
4. 控制面生成发布计划
5. 渲染落地配置与入口转发配置
6. 下发到目标节点并记录 release / task / operation

### 7.4 分享与订阅链路

1. 控制面根据用户当前实际生效节点生成目标集合
2. 为每条目标线路输出节点级订阅与直连分享
3. 同时输出聚合订阅
4. 后端统一生成二维码 SVG

## 8. 当前架构形态

当前架构是一个典型的单机控制面 MVP：

- 后端：单体 Node.js 服务
- 运行方式：单进程集中承载 API、任务、探测、发布、分享与 Web Shell 会话
- 前端：无打包链的静态多页面控制台
- 持久化：JSON 文件
- 会话：进程内内存 session
- 部署：单机 Docker / Compose

当前更像：

- 一个边界已逐渐清晰的“单体控制面”

而不是：

- 可横向扩展的多实例平台
- 已数据库化的正式生产架构

### 当前架构优点

- 结构集中，开发速度快
- 对低配节点友好
- 核心链路已经完整串起来
- 便于继续演化业务语义

### 当前架构硬边界

- 只适合单机部署
- 不适合多实例并发写入
- 服务重启会丢失内存会话
- 不具备成熟的任务队列、告警、审计与回滚体系

## 9. 当前最重要的产品语义

### 9.1 必须承认的事实

当前系统已经明确包含两种不同链路：

- 业务链路：用户如何进入入口节点并到达落地节点
- 管理链路：控制面如何通过 SSH 接管节点

这两种语义不能再混用。

### 9.2 当前正确方向

当前代码方向已经在朝这个目标收敛：

- `networking.*` 只承载业务流量语义
- `management.*` / SSH 只承载控制面语义
- 探测结果正在区分管理链路与业务链路
- 分享/订阅开始以发布结果和线路解析为准，而不只看节点 facts

### 9.3 仍需持续验证的点

- relay 线路在不同 profile / 入口端口组合下的稳定性
- 分享地址与实际业务入口的一致性
- 发布计划按入口节点聚合后的冲突校验
- 入口机 L4 转发与落地 sing-box 的长期可维护性

## 10. 当前已有 UI，但底层仍是 MVP 的部分

以下模块“不是没有”，但也不能被误判为“已经完全成熟”：

- Web Shell：已可用，但仍不是生产级 bastion
- 任务中心：已能看、能重试、能复探，但还缺真正完整的任务详情与调度策略
- 发布中心：已能真实发布，但协议深度、异常回滚与多策略编排仍有限
- 厂商页面：当前更像占位与统计看板，不是自动建机平台
- 路由拓扑：页面可展示，但复杂线路治理与冲突处理仍在演进
- 订阅与分享：已经有后端输出，但还需要持续校验“线路入口语义是否总是正确”

## 11. 当前主要风险与约束

### 11.1 架构层风险

- JSON 持久化仅适合单机 MVP
- session 在内存中，重启后会失效
- 没有正式的异步队列中间件
- 没有系统级审计、告警与回滚控制面
- API、任务、探测、执行与分享都在同一进程内，单点压力和恢复边界比较集中

### 11.2 产品层风险

- 当前定位若继续对外描述成“通用多云平台”，会误导开发优先级
- relay / 入口 / 落地 / SSH 跳板这几种概念一旦混写，后续会持续增加复杂度
- UI 范围扩展得较快，但部分底层能力还在 MVP 阶段

### 11.3 运维层风险

- 宿主机、容器、控制面、反向代理之间的环境变量一致性需要持续维护
- SSH 私钥、入口地址、节点真实 ingress 信息如果填错，会直接影响纳管与分享
- 当前仍缺少正式的告警通知、自愈闭环和资源替换策略

## 12. 当前不在范围内

以下内容当前不应被当成“已完成需求”：

- 多实例高可用控制面
- 数据库化正式架构
- 自动扩缩容与云厂商真实建机
- 完整面板 / NMS 适配闭环
- Clash / Mihomo 等多订阅格式输出
- 面向终端用户的自助门户
- 生产级堡垒机能力

## 13. 建议的下一阶段目标

### 目标 A：把单机控制面做稳

优先级：最高

建议内容：

- 数据层从 JSON 向数据库演进
- 会话与任务状态从进程内状态迁移到可持久化存储
- 完整化部署、备份、恢复、升级与回滚路径

完成标志：

- 服务重启不再影响登录态与关键状态
- 控制面具备稳定升级路径
- 单机生产可维护性明显提升

### 目标 B：把业务链路语义做透

优先级：高

建议内容：

- 继续收紧 `management` 与 `networking` 的边界
- 强化线路解析器与入口端口冲突校验
- 让发布、分享、探测统一依赖同一套业务路由解析结果

完成标志：

- relay 分享地址不再出现“发了落地机地址给用户”的歧义
- 同一入口机多线路时能明确冲突并拒绝错误发布
- 探测、发布、分享看到的是同一条线路事实

### 目标 C：把自动化闭环做起来

优先级：中高

建议内容：

- 增加告警和失败趋势
- 增加可配置的自动处置动作
- 再进入云厂商 API 自动建机 / 自愈替换

完成标志：

- 控制面能发现异常并触发明确动作
- 异常节点可被标记、隔离、替换
- 扩缩容不再只是路线图，而进入真实执行阶段

## 14. 本文结论

当前项目方向没有跑偏，但必须基于现实来描述：

- 它已经是“轻量代理节点控制面”，不是纯纳管台账
- 它已经具备真实主链路，不应再按 mock 项目看待
- 它仍然是单机 MVP，不应误判为最终生产架构
- 现在最重要的不是继续堆页面，而是把单机稳定性、链路语义和自动化闭环逐步做实
